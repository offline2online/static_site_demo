Menu Board API Reference

The Menu Board system stores all data in browser localStorage. These API specifications describe both the localStorage key/value schema (for direct integration) and a proposed REST API layer that can be implemented server-side as a thin bridge to the same data model. All endpoints use JSON payloads.

Introduction

The Menu Board is a browser-based system. Data lives in localStorage on each device. The REST API specifications below describe the contract for a server-side relay service that syncs data to/from localStorage across devices. Base URL for all endpoints: 

https://api.your-domain.com/v1/menuboard
ℹ️ All requests must include Content-Type: application/json and an Authorization: Bearer <token> header. The localStorage bridge implementation is described in the localStorage Bridge section.

Data Storage Keys

The following localStorage keys are used by the system:

KeyTypeDescription
mb_itemsJSON ArrayAll menu item objects
mb_settingsJSON ObjectRestaurant settings, logo, colours, orientation
mb_categoriesJSON Array<string>Category name strings
mb_stock_{STORECODE}JSON ObjectPer-store out-of-stock flags keyed by SKU
mb_last_updatedISO 8601 stringTimestamp of last data change — menu boards poll this to detect updates
mb_menu_skus_{type}JSON Array<string> | nullOptional SKU whitelist for a specific menu type. When set, only listed SKUs appear on that menu.

Product Item Schema

Full schema for a product item object:

Product Object
{
  "id":          "lxk3m2abc9",          // Internal ID (base36 + random)
  "sku":         "BRK001",              // Unique product SKU — used for external references
  "name":        "Classic Eggs Benedict",// Display name
  "description": "Poached eggs on toasted English muffin with hollandaise sauce",
  "category":    "Mains",               // Must match a value in mb_categories
  "price":       16.50,                 // Decimal price (no currency symbol)
  "active":      true,                  // false = hidden from all menu boards
  "menuTypes":   ["breakfast", "specials"],// Values: breakfast|lunch|dinner|drinks|specials
  "images":      ["data:image/png;base64,..."], // Array of base64 data URIs; first = primary
  "calories":    520,                   // null if not provided
  "fat":         28.5,                  // grams, null if not provided
  "carbs":       34.0,                  // grams, null if not provided
  "protein":     22.0,                  // grams, null if not provided
  "createdAt":   "2026-02-22T10:00:00.000Z",
  "updatedAt":   "2026-02-22T11:30:00.000Z"
}

Menu Control API

An external party (e.g. a POS system, campaign manager, or digital signage CMS) can push a list of product SKUs to control exactly which items appear on a specific menu type. When a SKU whitelist is set for a menu type, only those SKUs will be shown on that menu board (provided they are also active). Clearing the list restores the default (all active items for that menu type).

POST /menu/{type}/skus Set the SKU whitelist for a menu type
Path Parameters
ParameterTypeRequiredDescription
typestringRequiredMenu type: breakfast | lunch | dinner | drinks | specials
Request Body
FieldTypeRequiredDescription
skusstring[]RequiredOrdered array of SKUs to show on this menu. Order determines display order.
storestringOptionalStore code. If omitted, applies globally to all stores.
Example Request — Set Today's Specials
POST /v1/menuboard/menu/specials/skus
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...

{
  "skus": [
    "BRK001",
    "LCH005",
    "DRK012",
    "DSR003"
  ],
  "store": "STORE001"
}
200 OK
{
  "success": true,
  "menu": "specials",
  "store": "STORE001",
  "skus": ["BRK001", "LCH005", "DRK012", "DSR003"],
  "count": 4,
  "message": "Specials menu whitelist updated. Menu boards will refresh within 5 seconds."
}
⚠️ SKUs not found in the product catalogue are silently ignored. Only existing, active products matching the SKU whitelist will appear on the menu board.
200 OK 400 Invalid menu type

localStorage Bridge Implementation

For a lightweight integration without a server, external systems can write directly to localStorage on the menu board device using this JavaScript bridge. Call this from a CMS, kiosk controller, or parent frame.

Inject via postMessage (cross-origin)
// In the controlling CMS / parent window:
menuBoardIframe.contentWindow.postMessage({
  type: 'MB_SET_MENU_SKUS',
  menu: 'specials',
  skus: ['BRK001', 'LCH005', 'DRK012']
}, 'https://your-menu-board-domain.com');

// In menu-board.html (already listens for this):
window.addEventListener('message', (e) => {
  if (e.data.type === 'MB_SET_MENU_SKUS') {
    localStorage.setItem('mb_menu_skus_' + e.data.menu, JSON.stringify(e.data.skus));
    localStorage.setItem('mb_last_updated', new Date().toISOString());
    // Menu board auto-detects the change within 5 seconds
  }
});
Direct localStorage (same origin)
// Push a product update from a same-origin app:
function pushProductUpdate(sku, fields) {
  const items = JSON.parse(localStorage.getItem('mb_items') || '[]');
  const idx = items.findIndex(i => i.sku === sku);
  if (idx === -1) return false;
  items[idx] = { ...items[idx], ...fields, updatedAt: new Date().toISOString() };
  localStorage.setItem('mb_items', JSON.stringify(items));
  localStorage.setItem('mb_last_updated', new Date().toISOString());
  return true;
}

// Example: mark BRK001 as out of stock at store STORE001
const stock = JSON.parse(localStorage.getItem('mb_stock_STORE001') || '{}');
stock['BRK001'] = { outOfStock: true, updatedAt: new Date().toISOString() };
localStorage.setItem('mb_stock_STORE001', JSON.stringify(stock));
localStorage.setItem('mb_last_updated', new Date().toISOString());

Change Webhooks

Configure a webhook URL in settings to receive a POST notification whenever product data changes. This allows external systems to stay in sync without polling.

Webhook Payload (sent on every data change)
POST https://your-cms.example.com/webhook/menuboard
Content-Type: application/json
X-MenuBoard-Secret: <your-shared-secret>

{
  "event":       "items.updated",       // items.created | items.updated | items.deleted | stock.changed
  "timestamp":   "2026-02-22T12:00:00.000Z",
  "affected_skus": ["BRK001"],
  "store":       null                    // store code for stock events, null for global changes
}
💡 Menu boards poll mb_last_updated every 5 seconds. Any write to this key (from any source) causes all connected menu boards on that device to re-render immediately.

Stock API

The Stock API lets external systems (POS, ERP, warehouse management) read and update per-store stock levels and availability flags in real time. Stock data is stored in Firestore under stock/{STORE_CODE} and is read directly by the Retail Admin and Menu Board displays.

🔑 All Stock API requests must include the X-Stock-Api-Key header. Set your API key in HQ Admin → Settings → Stock API. In production, enforce this header via Firestore Security Rules — the demo uses open rules but the key is stored in menuboard/settings.stockApiKey for reference.
Authentication Header
X-Stock-Api-Key: {your-api-key-from-hq-settings}

Stock document data model — one document per store at stock/{STORE_CODE}:

Stock Document Schema
{
  "BRK001": {
    "quantity":   24,                          // Current estimated stock level (null = unknown)
    "outOfStock": false,                       // true = OOS, hides from menu board
    "notStocked": false,                       // true = store never carries this SKU
    "updatedAt":  "2026-03-11T09:15:00.000Z",  // ISO 8601 timestamp of last change
    "updatedBy":  "staff@example.com"          // Staff ID who made the last change
  },
  "LUN005": {
    "quantity":   null,
    "outOfStock": true,
    "notStocked": false,
    "updatedAt":  "2026-03-10T14:22:00.000Z",
    "updatedBy":  "staff@example.com"
  }
}
GET /stock/{store} Read full stock status for a store

Returns all SKU stock records for the given store. Uses the Firestore REST API directly — replace {PROJECT_ID} and {FIREBASE_API_KEY} with your project values.

PATCH /stock/{store} Bulk-update stock levels for multiple SKUs

Write stock data for one or more SKUs in a single request. Uses Firestore's PATCH with updateMask to perform a partial update — only the specified SKU fields are overwritten. All other SKUs in the store document remain unchanged.

Path Parameters
ParameterTypeRequiredDescription
storestringRequiredStore code (e.g. STORE001)
Request Body Fields
FieldTypeRequiredDescription
quantityintegerOptionalCurrent stock level. Set to null to clear.
outOfStockbooleanOptionalWhen true, hides item from menu board.
notStockedbooleanOptionalWhen true, marks as never stocked at this store.
updatedBystringOptionalIdentifier of the system or user making the change.
updatedAtstringOptionalISO 8601 timestamp of the change. Defaults to server time if omitted.
Example Request — update two SKUs
curl -X PATCH \
  "https://firestore.googleapis.com/v1/projects/{PROJECT_ID}/databases/(default)/documents/stock/STORE001?updateMask.fieldPaths=BRK001&updateMask.fieldPaths=LUN005&key={FIREBASE_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "X-Stock-Api-Key: your-api-key" \
  -d '{
    "fields": {
      "BRK001": {
        "mapValue": { "fields": {
          "quantity":   { "integerValue": "18" },
          "outOfStock": { "booleanValue": false },
          "notStocked": { "booleanValue": false },
          "updatedAt":  { "stringValue": "2026-03-11T10:00:00.000Z" },
          "updatedBy":  { "stringValue": "pos-system" }
        }}
      },
      "LUN005": {
        "mapValue": { "fields": {
          "quantity":   { "integerValue": "0" },
          "outOfStock": { "booleanValue": true },
          "notStocked": { "booleanValue": false },
          "updatedAt":  { "stringValue": "2026-03-11T10:00:00.000Z" },
          "updatedBy":  { "stringValue": "pos-system" }
        }}
      }
    }
  }'
200 OK — Success Response
{
  "name": "projects/{PROJECT_ID}/databases/(default)/documents/stock/STORE001",
  "fields": { /* full updated document returned */ },
  "updateTime": "2026-03-11T10:00:00Z"
}
200 OK 400 Bad Request — malformed field data 403 Forbidden — invalid or missing API key
POST /stock/{store}/sale Decrement quantity by 1 (record a sale)

Atomically decrements a SKU's quantity by 1 using a Firestore field transform. The quantity will not go below 0. If the resulting quantity reaches 0, outOfStock is automatically set to true by the Retail Admin listener — for POS-only integrations, set outOfStock explicitly via PATCH if needed.

Path Parameters
ParameterTypeRequiredDescription
storestringRequiredStore code (e.g. STORE001)
Request Body
FieldTypeRequiredDescription
skustringRequiredSKU of the product sold (e.g. BRK001)
Example Request
curl -X POST \
  "https://firestore.googleapis.com/v1/projects/{PROJECT_ID}/databases/(default)/documents:commit?key={FIREBASE_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "X-Stock-Api-Key: your-api-key" \
  -d '{
    "writes": [{
      "transform": {
        "document": "projects/{PROJECT_ID}/databases/(default)/documents/stock/STORE001",
        "fieldTransforms": [{
          "fieldPath": "BRK001.quantity",
          "increment": { "integerValue": "-1" }
        }]
      }
    }]
  }'
200 OK — Success Response
{
  "writeResults": [{
    "transformResults": [{ "integerValue": "17" }],   // new quantity after decrement
    "updateTime": "2026-03-11T10:05:00Z"
  }],
  "commitTime": "2026-03-11T10:05:00Z"
}
200 OK 400 Bad Request — SKU field not found or invalid 403 Forbidden — invalid or missing API key
POST /stock/{store}/return Increment quantity by 1 (record a return or restock)

Atomically increments a SKU's quantity by 1 using a Firestore field transform. Use this for customer returns, void transactions, or adding individual units back to available stock. If quantity rises above 0 and the item was previously marked OOS, clear outOfStock via a subsequent PATCH or let the Retail Admin auto-correct on the next stock sync.

Path Parameters
ParameterTypeRequiredDescription
storestringRequiredStore code (e.g. STORE001)
Request Body
FieldTypeRequiredDescription
skustringRequiredSKU of the product returned (e.g. BRK001)
Example Request
curl -X POST \
  "https://firestore.googleapis.com/v1/projects/{PROJECT_ID}/databases/(default)/documents:commit?key={FIREBASE_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "X-Stock-Api-Key: your-api-key" \
  -d '{
    "writes": [{
      "transform": {
        "document": "projects/{PROJECT_ID}/databases/(default)/documents/stock/STORE001",
        "fieldTransforms": [{
          "fieldPath": "BRK001.quantity",
          "increment": { "integerValue": "1" }
        }]
      }
    }]
  }'
200 OK — Success Response
{
  "writeResults": [{
    "transformResults": [{ "integerValue": "19" }],   // new quantity after increment
    "updateTime": "2026-03-11T10:08:00Z"
  }],
  "commitTime": "2026-03-11T10:08:00Z"
}
